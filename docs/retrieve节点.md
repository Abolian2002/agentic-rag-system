# Retrieve 节点详解（面试学习笔记）

本文档详细讲解 retrieve 节点中的核心检索技术，重点聚焦 BM25 稀疏检索的打分逻辑和混合检索的设计思路。

---

## 一、BM25 打分的四步骤

BM25 是一个基于词频和逆文档频率的文档打分算法，核心思想：**问题里的每个词，在文档中出现得越多、越稀有，文档就越相关**。

### 步骤 1：拆词（Tokenization）

把用户问题拆成关键词。

**示例问题**：`"LangGraph 的 StateGraph 怎么用？"`

**拆词结果**：`LangGraph`、`StateGraph`、`怎么`、`用`

---

### 步骤 2：算每个词的"重要度"（IDF）

不是所有词都一样重要——出现频率低的词，信息量大，区分度高。

| 词 | 全库出现频率 | 重要度 | 原因 |
|----|-------------|--------|------|
| `LangGraph` | 只在少数文档出现 | **高** | 专有名词，区分度高 |
| `StateGraph` | 只在少数文档出现 | **高** | 专有名词，区分度高 |
| `怎么` | 几乎所有文档都有 | **低** | 太常见，无区分度 |
| `用` | 几乎所有文档都有 | **低** | 太常见，无区分度 |

这就是 **IDF（Inverse Document Frequency，逆文档频率）** 的核心思想：

> **一个词在全库中出现得越少，它的"信息量"越大，权重越高。**

---

### 步骤 3：看词在目标文档里的出现次数（TF）

对每个候选文档，统计关键词出现的次数。

**示例**：

| 文档 | `LangGraph`出现次数 | `StateGraph`出现次数 | `怎么`出现次数 | `用`出现次数 |
|------|---------------------|----------------------|----------------|--------------|
| 文档A | 15次 | 8次 | 2次 | 3次 |
| 文档B | 2次 | 1次 | 5次 | 6次 |
| 文档C | 0次 | 0次 | 1次 | 2次 |

很明显，文档A中"关键的词"出现最多，应该排第一。

这就是 **TF（Term Frequency，词频）** 的核心思想：

> **一个词在某文档中出现得越多，这个文档越可能是答案。**

---

### 步骤 4：BM25 把 TF 和 IDF 结合起来打分

最终公式（简化版）：

```
分数 = Σ（每个关键词的 TF × IDF）
```

BM25 在此基础上做了两个数学优化：

**优化1：词频饱和（Saturation）**

词频不是无限增长的——出现100次和出现1000次差别不大，用一个饱和函数控制增量收益递减。

**优化2：文档长度归一化**

文档太长会稀释关键词密度，用长度归一化修正，让长文档和短文档在同等关键词密度下有公平的分数。

---

## 二、BM25 的两个关键参数

### k1 — 词频饱和参数

控制"词出现次数"对分数的影响强度：

| k1 值 | 效果 |
|-------|------|
| k1 = 0 | 不考虑词频，只看是否出现过（相当于布尔检索） |
| k1 = 1.2 ~ 2.0（常用） | 词频有影响但会饱和（出现很多次后增量收益递减） |
| k1 越大 | 词频影响越明显 |

### b — 文档长度归一化参数

控制文档长度对分数的影响：

| b 值 | 效果 |
|------|------|
| b = 0 | 完全不考虑文档长度 |
| b = 0.75（常用） | 中等程度归一化 |
| b = 1 | 完全归一化（长文档需要更多关键词才能达到相同分数） |

## 三、为什么叫"稀疏"检索？

和向量检索（稠密检索）对比：

| 维度 | BM25（稀疏） | 向量检索（稠密） |
|------|-------------|-----------------|
| 表示方式 | 词频向量——大部分维度是0（只有出现的词有值） | 语义向量——每个维度都有值（实数） |
| 匹配方式 | 精确词匹配 | 语义相似度匹配（余弦相似度） |
| 优点 | 精确、快、对专有名词/代码/数字效果好 | 能处理同义词、释义、概念级匹配 |
| 缺点 | 只能匹配完全相同的词（"苹果手机"找不到"iPhone"） | 对精确关键词、代码片段效果差 |

**"稀疏"这个词的来源**：BM25 的向量表示中，绝大多数位置是0（因为每个文档只包含词汇表中很小一部分词），所以是"稀疏向量"。

---

## 四、为什么需要混合检索（BM25 + 向量）？

这就是 **Hybrid Search（混合检索）** 的设计思路——取长补短：

- **BM25 负责"硬匹配"**：专有名词、代码函数名、精确术语、数字编号
- **向量检索负责"软匹配"**：同义词、改写、概念、语义理解

**示例对比**：

| 问题类型 | BM25 效果 | 向量检索效果 |
|----------|----------|-------------|
| `"LangChain 中的 RunnableMap 怎么用？"` | ✅ 精确匹配到包含 `RunnableMap` 的文档 | ❌ 可能找不到精确函数名 |
| `"怎么在 LangChain 里并行执行多个组件？"` | ❌ 找不到（没有 RunnableMap 这个词） | ✅ 能理解这和 `RunnableMap` 相关 |

**本系统的设计**：

通过 `EnsembleRetriever` 加权融合，权重配置：
- BM25 权重：**0.35**
- 向量检索权重：**0.65**

权重分配原因：语义匹配在 RAG 场景下更重要，但 BM25 的精确匹配能补向量检索的短板。

## 五、TF 和 IDF 的英文全称与计算时机

### 英文全称

- **TF = Term Frequency**（词频）
- **IDF = Inverse Document Frequency**（逆文档频率）

### 计算时机

TF 和 IDF **不是预先算好的固定值**，而是**查询时实时计算**——但需要在文档加载时构建预统计信息。

| 量 | 什么时候算 | 依赖什么 |
|----|-----------|---------|
| 词在每个文档中的出现次数 | 文档加载时（索引阶段）预先统计 | 单个文档内容 |
| 每个文档的总词数 | 文档加载时预先统计 | 单个文档内容 |
| 全库文档总数 N | 文档加载时统计 | 整个文档库 |
| 每个词在多少文档中出现（df） | 文档加载时统计 | 整个文档库 |
| **IDF（逆文档频率）** | **查询时实时算** | 全库统计量 |
| **TF（词频）** | **查询时实时算** | 文档级统计量 + 查询词 |

## 六、文档加载时构建什么？

当 PDF/DOCX 被切分成 chunks 并加载进 BM25 检索器时，系统会构建一个 **倒排索引（Inverted Index）**。

**结构示意**：

```
词汇表 → 出现这个词的所有文档ID + 每个文档里的出现次数
```

**示例**：

| 词 | 哪些文档有这个词 | 每个文档里出现几次 |
|----|----------------|-------------------|
| `LangGraph` | doc_1, doc_3, doc_7 | doc_1:15次, doc_3:8次, doc_7:2次 |
| `StateGraph` | doc_1, doc_7 | doc_1:10次, doc_7:3次 |
| `怎么` | doc_1, doc_2, doc_3, ... | ...（太多，权重低） |

同时记录：
- 每个文档的总词数（用于长度归一化）
- 全库文档总数 N
- 每个词在多少文档中出现（用于 IDF）

---

## 七、查询时发生了什么？

当用户问 `"LangGraph 的 StateGraph 怎么用"` 时：

1. **拆词** → `LangGraph`, `StateGraph`, `怎么`, `用`

2. **实时算 IDF**：
   - 对每个词，查倒排索引看它在多少文档中出现
   - 公式：`IDF(词) = log( (N - df + 0.5) / (df + 0.5) + 1 )`
   - 其中 N = 全库文档数，df = 该词出现的文档数

3. **实时算 TF**：
   - 对每个候选文档，查倒排索引看该词在文档中出现几次
   - 公式：`TF(词, 文档) = (出现次数 × (k1 + 1)) / (出现次数 + k1 × (1 - b + b × 文档长度/平均长度))`

4. **组合打分**：`分数 = Σ(TF × IDF)`，对所有候选文档排序

---

## 八、面试一句话总结

> BM25 在索引阶段构建倒排索引，记录每个词在哪些文档中出现、出现多少次、每个文档多长。真正的 TF 和 IDF 分数是在查询时基于这些预统计数据实时计算的——因为 IDF 依赖全库文档分布，TF 依赖具体查询词，只有查询来了才能算出最终分数。

## 九、Retrieve 节点的完整管线

BM25 只是 retrieve 节点的第一阶段。完整管线如下：

| 阶段 | 技术 | 执行方式 | 职责 |
|------|------|---------|------|
| 1 | BM25 稀疏检索 | 并行 | 精确词匹配，抓专有名词/代码 |
| 2 | Chroma 稠密向量检索 | 并行 | 语义匹配，抓同义词/概念 |
| 3 | Ensemble 融合 | 串行 | 加权合并两路结果（0.35/0.65） |
| 4 | GraphRAG（可选） | 并行/串行 | 实体共现图补充跨段落关联 |
| 5 | CrossEncoder 重排（可选） | 串行 | 对 Top-K 神经网络精排 |
| 6 | CRAG 分级（可选） | 串行 | LLM 逐一判断相关性，低置信度触发重试 |

**一句话记忆**：检索阶段做并行（BM25 + Chroma），排序过滤阶段做串行（重排 + CRAG）——先尽快把"可能相关"的文档捞出来，再逐步精筛。

## 十、关键面试 Q&A

### Q1：BM25 和 TF-IDF 有什么区别？

BM25 是 TF-IDF 的改进版本。主要区别：
- TF-IDF 的 TF 是线性增长（出现次数直接乘）
- BM25 的 TF 有饱和函数（出现很多次后收益递减）
- BM25 加入文档长度归一化（长文档不会天然占优势）

### Q2：为什么 BM25 权重只给 0.35？

在 RAG 场景下，用户的自然语言问题往往和文档措辞不一致，语义匹配更重要。但 BM25 对专有名词、代码函数名的精确匹配不可替代，所以保留一定权重做补充。

### Q3：倒排索引和向量索引的区别？

| 索引类型 | 存储内容 | 查询方式 |
|----------|---------|---------|
| 倒排索引 | 词 → 文档列表 + 词频 | 精确词匹配，O(1) 查词 |
| 向量索引 | 文档 → embedding 向量 | 近似最近邻搜索（ANN），O(log N) 或 O(N) |

倒排索引是 BM25 的基础，向量索引是稠密检索的基础。混合检索就是两套索引并行查，再融合结果。
