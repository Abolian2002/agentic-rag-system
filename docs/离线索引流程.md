# 离线索引流程详解（面试学习笔记）

## 一、完整流程总览

```
用户上传文件 → 解析 → 清洗 → 分块 → 绑定元数据 → 向量化 → 存 Chroma → 建 BM25 → 可选建图 → 计算 Hash
```

## 二、9 个步骤详解

### 步骤 1：解析（Parsing）

把不同格式的文件转成纯文本。

**支持的文件格式**：

| 格式 | 解析方式 | 说明 |
|------|---------|------|
| **PDF** | PyPDFLoader | 提取文本内容（不含图片） |
| **DOCX** | Docx2txtLoader | 提取文本，保留基本格式 |
| **TXT** | 直接读取 | 纯文本文件 |
| **Markdown** | UnstructuredMarkdownLoader | 保留标题结构 |

**解析过程**：
1. 根据扩展名识别文件类型
2. 调用对应 Loader 读取文件
3. 合并分段，输出完整文本

---

### 步骤 2：清洗（Cleaning）

去掉文本中的噪声，提高分块质量。

| 步骤 | 操作 | 示例 |
|------|------|------|
| **去空白** | 去掉连续换行、空格、制表符 | `"  \n\n\n  LangGraph\n  "` → `"LangGraph"` |
| **去页眉页脚** | 去掉重复出现的页眉（如"第X页"） | `"第5页\nLangGraph\n第5页"` → `"LangGraph"` |
| **去特殊字符** | 去掉不可见字符、乱码、控制字符 | `"LangGraph\x00框架"` → `"LangGraph框架"` |
| **合并段落** | 把被错误拆分的段落合并 | `"LangGraph 是\n一个框架"` → `"LangGraph 是一个框架"` |
| **去重复** | 去掉重复的内容块 | 同一文档中重复的版权声明 |

**为什么清洗很重要？**
不清洗会导致：连续换行导致分块断裂、页眉页脚干扰检索相关性、特殊字符影响 embedding 质量。

---

### 步骤 3：分块（Chunking）

把长文本切成适合检索的小块。

**为什么要分块？**

| 问题 | 说明 |
|------|------|
| 模型上下文限制 | LLM 有 token 限制，不能一次喂太长文本 |
| 检索精度 | 小块比大块更容易匹配到具体问题 |
| 相关性 | 小块的 embedding 更精准 |

**分块策略：递归分块**

```
1. 先按大分隔符切（段落 "\n\n"）
2. 太大 → 再按换行切（"\n"）
3. 还太大 → 按空格切（" "）
4. 还太大 → 按字符硬切
```

**关键参数**：

| 参数 | 值 | 说明 |
|------|-----|------|
| **chunk_size** | 512 | 每个 chunk 的最大字符数 |
| **chunk_overlap** | 50 | 相邻 chunk 重叠的字符数 |
| **length_function** | len | 用字符数衡量长度 |

**chunk_overlap 的作用**：防止一句话被拆到两个 chunk 里导致语义断裂。

```
无重叠：
chunk1: "LangGraph 是一个状态图框架，"
chunk2: "用于构建有状态的工作流。"

有重叠（overlap=10）：
chunk1: "LangGraph 是一个状态图框架，用于"
chunk2: "用于构建有状态的工作流。"
```

**特殊格式处理**：
- Markdown：按标题级别切（#、##、###），保证章节完整性
- 代码文件：按函数/类切，保证代码块完整性

---

### 步骤 4：绑定元数据（Metadata）

每个 chunk 绑定溯源信息，方便定位和过滤。

| 字段 | 内容 | 用途 |
|------|------|------|
| **source** | 来源文件名 | 告诉用户答案来自哪个文件 |
| **chunk_id** | chunk 的序号 | 定位具体位置 |
| **total_chunks** | 总 chunk 数 | 了解文档结构 |
| **start_index** | 在原文档中的起始位置 | 定位具体段落 |
| **end_index** | 在原文档中的结束位置 | 定位具体段落 |

---

### 步骤 5：向量化（Embedding）

把每个 chunk 转成高维向量。

```
chunk 文本 → Ollama → nomic-embed-text 模型 → 768维向量
```

**为什么用 nomic-embed-text？**

| 原因 | 说明 |
|------|------|
| 免费 | 本地运行，不花钱 |
| 支持中文 | 对中文语义理解好 |
| 维度适中 | 768维，够表达语义又不会太大 |
| 速度快 | 本地推理，延迟低 |

---

### 步骤 6：存入 Chroma 向量数据库

向量持久化存储，自动构建 ANN 索引。

**Chroma 自动做的事情**：
1. 创建 HNSW 近似最近邻索引
2. 持久化到磁盘（重启不丢）
3. 建立元数据倒排索引（用于过滤）

```python
vector_store = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./chroma_db",
    collection_name="rag_collection"
)
```

---

### 步骤 7：构建 BM25 稀疏索引

同步构建基于词频的稀疏检索索引。

**BM25 自动做的事情**：
1. 构建倒排索引（词 → 文档列表）
2. 统计词频、文档长度、文档总数
3. 预计算 IDF 相关统计量

```python
bm25_retriever = BM25Retriever.from_documents(chunks)
```

---

### 步骤 8：可选构建实体共现图（GraphRAG）

可选：构建基于实体共现的知识图谱。

**流程**：
1. 从每个 chunk 中提取实体（专有名词、技术术语）
2. 同一 chunk 中的实体之间建立边（共现关系）
3. 用 NetworkX 存储无向图

```python
graph = nx.Graph()
for chunk in chunks:
    entities = extract_entities(chunk)
    for i, j in combinations(range(len(entities)), 2):
        graph.add_edge(entities[i], entities[j])
```

---

### 步骤 9：计算 Hash（增量索引）

为每个文件计算指纹，用于增量更新。

**算法**：SHA-256

**特性**：
- 同一个文件 → 同一个 Hash
- 文件改一个字 → Hash 完全变
- 不同文件 → Hash 几乎不会碰撞

```python
file_hash = hashlib.sha256(file_content).hexdigest()
```

**增量索引原理**：

| 场景 | Hash 对比结果 | 操作 |
|------|-------------|------|
| 文件没变 | Hash 相同 | 跳过，不复用上次的索引 |
| 文件新增 | 旧 Hash 中没有 | 新索引这个文件 |
| 文件修改 | Hash 变了 | 删除旧的，重新索引 |
| 文件删除 | 当前文件中没有 | 删除对应的索引 |

---

## 三、增量索引工作原理

```
启动时：
1. 读取上次保存的 Hash 记录（JSON 文件）
2. 扫描当前目录下所有文件，计算每个文件的 Hash
3. 对比新旧 Hash：
   - 相同 → 跳过
   - 不同/新增 → 重新索引这个文件
   - 删除了 → 删除对应的索引
4. 更新 Hash 记录文件
```

## 四、面试怎么讲？（30秒版）

> 离线索引分为九步：首先解析不同格式的文件转成纯文本；然后做清洗——去空白、去页眉页脚、去特殊字符、合并段落；接着用递归分块策略切成 512 字符大小的 chunk，50 字符重叠保证语义连贯；每个 chunk 绑定来源文件、序号等元数据；然后通过本地 Ollama 的 nomic-embed-text 生成 768 维 embedding；存入 Chroma 向量库自动构建 HNSW 索引；同步构建 BM25 稀疏索引；可选构建 NetworkX 实体共现图；最后计算每个文件的 SHA-256 Hash 用于增量更新——下次启动只处理 Hash 变化的文件，不需要全量重建。

## 六、面试高频追问

### Q1：chunk_size 为什么选 512？

512 字符是经验值。太大则单个 chunk 包含的信息太多，embedding 不够精准；太小则语义不完整，检索回来也没法回答问题。512 字符大约 300-400 个汉字，刚好是一段完整的内容，既能保证语义完整，又能让 embedding 足够精准。

### Q2：chunk_overlap 为什么选 50？

overlap 太小则句子容易被切断；overlap 太大则冗余太多，浪费存储空间和计算资源。50 字符大约是一句话的长度，能保证重要句子不会被切断。

### Q3：为什么用递归分块而不是按字符硬切？

按字符硬切会把句子、段落从中间切断，破坏语义完整性。递归分块优先按段落、换行、空格这些天然分隔符切，尽量保证语义完整，只有实在太大时才按字符硬切。

### Q4：增量索引中 Hash 对比是怎么实现的？

启动时读取一个 JSON 文件，里面存着每个文件的文件名和对应的 SHA-256 Hash。然后扫描当前目录，对每个文件重新计算 Hash，和 JSON 里的对比——相同就跳过，不同就重新索引。最后更新 JSON 文件。这样就实现了只处理有变化的文件。

## 核心概念速记

| 术语 | 一句话解释 |
|------|-----------|
| 解析 | 把不同格式的文件转成纯文本 |
| 清洗 | 去掉文本中的噪声（空白、页眉、特殊字符） |
| 分块 | 把长文本切成适合检索的小块 |
| 递归分块 | 优先按大分隔符切，太大再按小分隔符切 |
| chunk_overlap | 相邻 chunk 重叠的字符数，保证语义连贯 |
| 元数据 | chunk 的溯源信息（来源文件、位置等） |
| 向量化 | 把文本转成高维向量（embedding） |
| SHA-256 | 哈希算法，用于计算文件指纹 |
| 增量索引 | 只处理有变化的文件，不全量重建 |
