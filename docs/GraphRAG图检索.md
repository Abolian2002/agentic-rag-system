# GraphRAG 图检索详解（面试学习笔记）

本文档详细讲解 retrieve 节点中的 GraphRAG（基于图的检索增强）技术，包括核心概念、图的构建步骤、图的查询步骤、项目配置和面试要点。

---

## 一、什么是 GraphRAG？

**GraphRAG = Graph-based RAG（基于图的 RAG）**

核心思想：**文档不只是孤立的文本块，它们之间有语义关联**。

传统向量检索只看"这段文字和问题像不像"，但忽略了文档之间的关系——比如"LangGraph"和"LangChain"在多个文档中同时出现，说明它们是相关的。

GraphRAG 用图结构来捕捉这些关联，当检索到一个文档时，还能沿着图找到它的"邻居"文档，补充更多相关信息。

---

## 二、为什么需要 GraphRAG？

### 传统 RAG 的局限

| 场景 | 传统向量检索 | 问题 |
|------|------------|------|
| 问"LangGraph 是什么" | 只返回提到 LangGraph 的文档 | 可能漏掉讲 LangChain 和 LangGraph 关系的文档 |
| 问"如何安装 Python 依赖" | 只返回讲 pip 的文档 | 可能漏掉讲 uv、conda 的文档 |
| 问"Streamlit 怎么用" | 只返回讲 Streamlit 的文档 | 可能漏掉讲 FastAPI + Streamlit 结合的文档 |

**GraphRAG 的解决方案**：建立文档之间的关联图，当检索到一个文档时，沿着图"走"几步，找到相关的邻居文档。

---

## 三、图的构建（索引阶段，三步）

### 步骤 1：实体提取

从每个文档 chunk 中提取实体（专有名词、人名、技术术语等）。

**提取方式**：基于 NLP 的名词短语提取（词性标注）。

**示例**：
- 文档内容："LangGraph 是 LangChain 团队开发的状态图框架..."
- 提取实体：`LangGraph`、`LangChain`、`状态图框架`

**为什么要提取实体？**

实体是文档内容的核心标识——"LangGraph"比"是"、"开发"这样的普通词更有信息量，更容易建立有意义的关联。

---

### 步骤 2：建立共现关系

如果两个实体出现在同一个 chunk 中，就在它们之间建立一条边。

**示例**：

```
chunk_1: "LangGraph 是 LangChain 团队开发的状态图框架..."
→ 边：LangGraph ↔ LangChain

chunk_2: "LangChain 提供了许多工具，包括 LangGraph..."
→ 边：LangChain ↔ LangGraph

chunk_3: "StateGraph 是 LangGraph 的核心概念..."
→ 边：StateGraph ↔ LangGraph
```

**共现的含义**：两个实体同时出现在一段文本中，说明它们在语义上是相关的。

---

### 步骤 3：构建共现图

把所有实体（节点）和边放到一张图里：

```
      ┌───────────┐
      │  LangChain│
      └─────┬─────┘
            │
      ┌─────▼─────┐
      │  LangGraph│
      └─────┬─────┘
            │
      ┌─────▼─────┐
      │StateGraph │
      └───────────┘
```

**图的结构**：
- **节点**：实体（专有名词、技术术语、人名等）
- **边**：共现关系（实体同时出现在一个 chunk 中）
- **无向图**：边没有方向（A 共现 B 等价于 B 共现 A）

**本项目用 NetworkX 实现**——一个轻量级的 Python 图库，适合个人知识库规模。
NetworkX 是 Python 中常用的内存图库，适合构建中小规模的图。它支持多种图类型，API 简洁，内置丰富的图算法。在我们的 GraphRAG 中，图规模不大（几百到几千个实体节点），只需要简单的图游走操作（找邻居），NetworkX 完全够用，而且开发成本低，不需要部署额外服务。

如果未来文档量大幅增长，图规模达到十万级以上，或者需要持久化、并发查询，就应该考虑迁移到专业的图数据库（如 Neo4j）。但目前的个人知识库场景，NetworkX 是最优选择。

**图的规模**：
- 节点数：和文档中的实体数相当（几百到几千个）
- 边数：和实体共现次数相当
- 平均度：每个实体连接的邻居数（通常 2-10）

## 四、图的查询（检索阶段，四步）

### 步骤 1：从问题中提取实体

用户问 `"LangGraph 的 StateGraph 怎么用？"` → 提取实体：`LangGraph`、`StateGraph`

---

### 步骤 2：在图中查找实体

找到这些实体在图中的位置：

```
      ┌───────────┐
      │  LangChain│ ← 间接关联
      └─────┬─────┘
            │
      ┌─────▼─────┐
      │  LangGraph│ ← 用户问题中的实体
      └─────┬─────┘
            │
      ┌─────▼─────┐
      │StateGraph │ ← 用户问题中的实体
      └───────────┘
```

**如果实体不在图中怎么办？**

跳过该实体，继续处理其他实体；如果所有实体都不在图中，GraphRAG 不返回任何结果。

---

### 步骤 3：图游走（Graph Traversal）

从这些实体出发，沿着边"走"几步，找到相关的邻居实体。

**本项目配置**：只走一步（找直接邻居）。

**为什么只走一步？**

- 步数太多会导致召回范围过大，引入噪音
- 步数太少则可能漏掉有用的关联
- 一步是经验平衡点

**示例**：
- 从 `LangGraph` 出发 → 找到 `LangChain`、`StateGraph`
- 从 `StateGraph` 出发 → 找到 `LangGraph`

---

### 步骤 4：根据邻居实体召回文档

找到包含这些邻居实体的文档，作为补充检索结果。

**示例**：
- 邻居实体 `LangChain` → 召回讲 LangChain 和 LangGraph 关系的文档
- 邻居实体 `StateGraph` → 召回讲 StateGraph 用法的文档

**召回方式**：在文档库中搜索包含邻居实体的 chunk。

---

## 五、项目中的实际配置

看 [knowledge_graph.py](file:///Users/scm/code/CORTEX-AI-SUPER-RAG/src/retrieval/knowledge_graph.py)：

```python
# 构建共现图
graph = nx.Graph()

# 添加节点（实体）和边（共现关系）
for chunk in chunks:
    entities = extract_entities(chunk)
    for i in range(len(entities)):
        for j in range(i + 1, len(entities)):
            graph.add_edge(entities[i], entities[j])

# 查询时的图游走
def query_graph(query, top_k=2):
    entities = extract_entities(query)
    related_docs = []
    for entity in entities:
        if entity in graph:
            # 找邻居实体
            neighbors = list(graph.neighbors(entity))[:top_k]
            # 根据邻居实体召回文档
            related_docs.extend(retrieve_by_entities(neighbors))
    return related_docs
```

### GraphRAG 在检索管线中的位置

```
BM25 + Chroma → Ensemble融合 → GraphRAG补充 → CrossEncoder重排 → CRAG分级
```

GraphRAG 的结果和 EnsembleRetriever 的结果合并，再一起送进 CrossEncoder 重排。

## 六、为什么项目中 GraphRAG 是可选的？

| 原因 | 说明 |
|------|------|
| **规模问题** | 个人知识库文档量不大时，向量检索已经足够 |
| **计算成本** | 构建图和图游走需要额外时间和内存 |
| **实体提取质量** | 如果实体提取不准，图的质量就差 |
| **简单够用** | 对大多数场景，BM25 + 向量检索已经能满足需求 |

**什么时候开启 GraphRAG？**
- 文档量较大（>1000 个 chunk）
- 文档之间有明显的关联关系（比如技术文档、学术论文）
- 用户经常问需要跨文档关联的问题

---

## 七、GraphRAG 和向量检索的区别

| 维度 | 向量检索 | GraphRAG |
|------|---------|----------|
| 匹配依据 | 语义相似度（余弦距离） | 实体共现关系（图结构） |
| 能找到 | 语义相近的文档 | 关联的文档（即使语义不相似） |
| 示例 | "LangGraph 用法"找到讲 LangGraph 的文档 | "LangGraph"找到讲 LangChain 的文档 |
| 互补性 | 强 | 强 |

**两者不是替代关系，而是互补关系**——向量检索负责语义匹配，GraphRAG 负责关联发现。

---

## 八、更复杂的 GraphRAG 形式

我们项目用的是**实体共现图**（最简单的形式），工业界还有更复杂的形式：

| 形式 | 思路 | 复杂度 |
|------|------|--------|
| **实体共现图** | 同一 chunk 中的实体连边 | 低（本项目用） |
| **知识图谱** | 实体之间有明确的关系类型（如"属于"、"开发于"） | 中 |
| **文档级图** | 文档之间基于相似度连边 | 中 |
| **HyperGraph** | 多个节点之间建立超边（更复杂的关系） | 高 |

## 九、面试怎么讲？（30秒版）

> GraphRAG 是基于图的检索增强，核心是建立文档之间的实体共现关系图。索引时从每个 chunk 中提取实体，同一 chunk 中的实体之间建立边，用 NetworkX 存储。查询时从问题中提取实体，在图中做游走找到邻居实体，再根据邻居实体召回相关文档。这样能补充向量检索遗漏的跨文档关联信息——比如问 LangGraph 时，不仅能找到讲 LangGraph 的文档，还能找到讲 LangChain 和 LangGraph 关系的文档。我们项目中 GraphRAG 是可选的，因为个人知识库规模不大时，BM25+向量已经足够。

## 十、面试高频追问

### Q1：为什么不用更复杂的知识图谱？

复杂知识图谱需要标注实体之间的关系类型（如"属于"、"开发于"），这需要大量人工标注或更复杂的 NLP 模型。个人项目用实体共现图已经能捕捉大部分关联关系，而且实现简单，不需要额外标注。

### Q2：实体提取怎么做？

项目中用的是简单的 NLP 方法——名词短语提取（基于词性标注）。在生产环境中，可以用更复杂的模型（如 spaCy 的 NER、BERT 实体识别）来提高精度。

### Q3：图游走的步数怎么定？

本项目只走一步（找直接邻居）。步数太多会导致召回范围过大，引入噪音；步数太少则可能漏掉有用的关联。一步是经验平衡点。

### Q4：GraphRAG 和 RAG-Fusion 有什么区别？

| 维度 | GraphRAG | RAG-Fusion |
|------|----------|------------|
| 原理 | 基于实体共现图的关联发现 | 基于多查询变体的结果融合 |
| 输入 | 问题中的实体 | 问题的多个改写版本 |
| 作用 | 补充跨文档关联信息 | 提高召回率和排序质量 |

两者可以结合使用——先用 RAG-Fusion 生成多查询变体，再用 GraphRAG 补充关联文档。

### Q5：如果文档量很大，NetworkX 够用吗？

不够用。NetworkX 是内存中的图，文档量上万时会占太多内存。这时候需要用专业的图数据库（如 Neo4j、NebulaGraph）来存储和查询。

## 十二、构建与查询步骤总结

### 图的构建（索引阶段）

| 步骤 | 名称 | 核心操作 | 为什么重要 |
|------|------|---------|-----------|
| 1 | 实体提取 | 从每个 chunk 提取专有名词、术语 | 实体是文档内容的核心标识 |
| 2 | 建立共现关系 | 同一 chunk 中的实体之间连边 | 共现意味着语义相关 |
| 3 | 构建共现图 | 用 NetworkX 存储所有节点和边 | 图结构便于高效游走查询 |

### 图的查询（检索阶段）

| 步骤 | 名称 | 核心操作 | 为什么重要 |
|------|------|---------|-----------|
| 1 | 提取问题实体 | 从用户问题中提取实体 | 找到图中的入口点 |
| 2 | 在图中查找实体 | 确认实体是否在图中 | 不在图中则跳过 |
| 3 | 图游走 | 从实体出发找邻居（一步） | 发现跨文档关联 |
| 4 | 召回关联文档 | 根据邻居实体召回文档 | 补充向量检索遗漏的信息 |

**一句话记忆**：GraphRAG = 建实体共现图 + 图游走找邻居 + 召回关联文档，补充向量检索的跨文档关联。
