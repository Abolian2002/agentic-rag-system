# Ensemble 加权融合详解（面试学习笔记）

本文档详细讲解 retrieve 节点中的 Ensemble 加权融合技术，包括核心概念、四步融合流程、项目配置和面试要点。

---

## 一、什么是 Ensemble 加权融合？

**Ensemble（集成）** 就是把多个检索器的结果合并起来，用权重加权，让最终排名比任何单个检索器都好。

**类比**：就像找工作时，HR 会综合笔试成绩（权重 0.4）和面试表现（权重 0.6）来决定录取谁——单一分数可能不准，加权平均更全面。

### 为什么需要融合？

| 检索器 | 优点 | 缺点 |
|--------|------|------|
| **BM25** | 精确匹配专有名词、代码、数字 | 无法处理同义词、改写 |
| **Chroma 向量** | 能理解语义、处理同义词 | 对精确关键词不如 BM25 |

**融合前的问题**：
- 只用 BM25：问"怎么在 LangChain 里并行执行组件"，找不到 `RunnableMap` 相关文档
- 只用向量：问"RunnableMap 怎么用"，可能漏了精确匹配的文档

**融合后的效果**：两者互补，召回率和准确率都提升。

---

## 二、加权融合的四步骤

### 步骤 1：各自检索

BM25 和 Chroma 各自返回 Top-K 文档（比如各返回 10 个），并带上自己的分数。

**示例**：

| 文档 | BM25 分数 | 向量相似度 |
|------|-----------|-----------|
| doc_A | 85 | 0.72 |
| doc_B | 70 | 0.85 |
| doc_C | 60 | 0.90 |
| doc_D | 90 | 0.60 |

**特点**：两个检索器并行执行，互不依赖。

---

### 步骤 2：分数归一化（关键步骤）

BM25 的分数范围是 [0, 100+]，向量相似度是 [0, 1]——**单位不一样，不能直接加**。

LangChain 的 `EnsembleRetriever` 会自动做归一化：
- **BM25 分数** → 线性映射到 [0, 1]
- **向量相似度** → 本身就在 [0, 1]，不用变

**归一化后**：

| 文档 | BM25（归一化） | 向量相似度 |
|------|---------------|-----------|
| doc_A | 0.85 | 0.72 |
| doc_B | 0.70 | 0.85 |
| doc_C | 0.60 | 0.90 |
| doc_D | 0.90 | 0.60 |

**为什么归一化是关键？**

如果不归一化，BM25 的分数（比如 85）远大于向量相似度（0.72），加权后结果几乎完全由 BM25 主导，融合就失去了意义。归一化保证两个检索器的分数在同一尺度上可比。

---

### 步骤 3：加权求和

```
最终分数 = w1 × BM25分数 + w2 × 向量相似度
```

本项目配置：
- w1 = 0.35（BM25 权重）
- w2 = 0.65（向量权重）

**计算示例**：

| 文档 | BM25(0.35) | 向量(0.65) | 最终分数 |
|------|------------|-----------|---------|
| doc_A | 0.85×0.35=0.2975 | 0.72×0.65=0.468 | **0.7655** |
| doc_B | 0.70×0.35=0.245 | 0.85×0.65=0.5525 | **0.7975** |
| doc_C | 0.60×0.35=0.21 | 0.90×0.65=0.585 | **0.795** |
| doc_D | 0.90×0.35=0.315 | 0.60×0.65=0.39 | **0.705** |

**为什么权重是 0.35/0.65？**

- 向量检索权重更高（0.65）：在 RAG 场景下，用户的自然语言问题往往和文档措辞不一致，语义匹配更重要
- BM25 保留一定权重（0.35）：对专有名词、代码函数名的精确匹配不可替代

---

### 步骤 4：去重排序

如果同一个文档被两个检索器都召回了，只保留一条，按最终分数排序。

**最终排名**：
1. doc_B（0.7975）
2. doc_C（0.795）
3. doc_A（0.7655）
4. doc_D（0.705）

**去重规则**：
- 按文档唯一标识（如 chunk_id）去重
- 保留分数最高的那条记录

## 三、项目中的实际配置

看 [retriever.py](file:///Users/scm/code/CORTEX-AI-SUPER-RAG/src/retrieval/retriever.py) 中的代码：

```python
ensemble_retriever = EnsembleRetriever(
    retrievers=[bm25_retriever, chroma_retriever],
    weights=[0.35, 0.65]
)
```

### 完整流程示意

```
BM25 检索 → Top-K 文档 + 分数
              ↓
         归一化 → [0,1]
              ↓
         加权求和 (0.35)
              ↓
最终排名 ← 去重排序
              ↑
         加权求和 (0.65)
              ↑
         归一化 → [0,1]
              ↑
Chroma 向量检索 → Top-K 文档 + 分数
```

## 四、其他融合策略对比

除了加权求和，还有其他融合方法：

| 策略 | 思路 | 公式 | 适用场景 |
|------|------|------|---------|
| **加权求和** | 分数 × 权重后相加 | w1·s1 + w2·s2 | 最常用，简单有效 |
| **RRF（Reciprocal Rank Fusion）** | 基于排名位置融合 | Σ 1/(k + rank) | RAG-Fusion 多查询场景 |
| **Concatenate** | 直接把结果拼起来 | - | 简单但可能有重复 |
| **Max** | 取两个分数的最大值 | max(s1, s2) | 只关心"是否命中" |

**本项目的选择**：
- 主检索管线用 **加权求和**（EnsembleRetriever）
- RAG-Fusion 多查询变体融合时用 **RRF**

## 五、面试怎么讲？（30秒版）

> 我们用了 `EnsembleRetriever` 做加权融合，把 BM25 和 Chroma 向量检索的结果合并。流程是：两个检索器各自返回 Top-K 文档，EnsembleRetriever 先做分数归一化（把不同尺度的分数统一到 [0,1]），然后按权重加权求和——BM25 权重 0.35，向量权重 0.65——最后去重排序。这样既能保留 BM25 对精确关键词的优势，又能利用向量检索的语义匹配能力，融合后的召回率比单独用任何一种都高。

## 六、面试高频追问

### Q1：为什么要做归一化？

如果不归一化，BM25 的分数（比如 85）远大于向量相似度（0.72），加权后结果几乎完全由 BM25 主导，融合就失去了意义。归一化保证两个检索器的分数在同一尺度上可比。

### Q2：权重怎么定的？

0.35/0.65 是经验值。我们做过对比测试：
- 向量权重太高（0.8+）：对代码、专有名词的精确匹配变差
- BM25 权重太高（0.5+）：对同义词、改写的召回下降
- 0.35/0.65 在两者之间取得平衡，在我们的文档集上效果最好

如果换一个文档集（比如以代码为主 vs 以自然语言为主），这个权重可能需要调整。

### Q3：和 RRF（Reciprocal Rank Fusion）有什么区别？

| 维度 | 加权求和 | RRF |
|------|---------|-----|
| 核心依据 | 分数本身 | 排名位置 |
| 对分数敏感 | 高（分数越高贡献越大） | 低（只看排名） |
| 适用场景 | 两个检索器融合 | 多查询变体融合 |
| 公式复杂度 | 低 | 中 |

RRF 更适合多查询融合（RAG-Fusion），因为每个查询的分数尺度可能不一样；加权求和适合两个检索器融合，因为我们可以控制权重。

### Q4：如果有三个检索器怎么办？

原理一样，扩展权重向量即可：
```python
ensemble_retriever = EnsembleRetriever(
    retrievers=[bm25, chroma, graph_rag],
    weights=[0.3, 0.5, 0.2]
)
```
关键是保证所有权重之和为 1。

## 七、核心概念速记

| 术语 | 一句话解释 |
|------|-----------|
| Ensemble | 集成，把多个检索器的结果合并 |
| 加权求和 | 分数 × 权重后相加 |
| 归一化 | 把不同尺度的分数统一到 [0,1] |
| 去重排序 | 移除重复文档，按最终分数排序 |
| RRF | 基于排名位置的融合方法 |

---

## 八、四步骤总结

| 步骤 | 名称 | 核心操作 | 为什么重要 |
|------|------|---------|-----------|
| 1 | 各自检索 | BM25 和向量分别召回 Top-K | 并行执行，互不依赖 |
| 2 | 分数归一化 | 统一到 [0,1] 尺度 | 保证分数可比，不被某一方主导 |
| 3 | 加权求和 | 按 0.35/0.65 权重计算最终分数 | 平衡精确匹配和语义匹配 |
| 4 | 去重排序 | 移除重复，按分数排序 | 输出唯一、有序的结果列表 |

**一句话记忆**：Ensemble 融合 = 各自检索 → 归一化 → 加权求和 → 去重排序，让 BM25 的精确和向量的语义互补。
