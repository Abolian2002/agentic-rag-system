# 分块策略详解（面试学习笔记）

## 一、常见分块策略对比

| 策略 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| **固定大小分块** | 按固定字符数切 | 简单、可控 | 可能切断语义 | 通用场景 |
| **递归分块** | 优先按分隔符切，太大再细分 | 尽量保持语义完整 | 实现稍复杂 | 通用场景（本项目用）|
| **语义分块** | 用 embedding 找语义边界 | 语义完整性最好 | 需要调用模型，慢 | 高质量文档 |
| **结构化分块** | 按标题、章节、代码块切 | 保持结构完整 | 需要解析结构 | Markdown、HTML |
| **段落分块** | 按段落切 | 天然语义边界 | 段落大小不一 | 新闻、论文 |
| **滑动窗口分块** | 固定步长滑动切 | 覆盖所有内容 | 冗余度高 | 需要覆盖所有细节 |
| **基于实体的分块** | 按实体边界切 | 适合知识密集型文档 | 需要实体识别 | 百科、技术文档 |
| **分层分块** | 粗分+细分结合 | 兼顾全局和细节 | 存储开销大 | 需要多粒度检索 |

## 二、详细讲解每种策略

### 1. 固定大小分块（Fixed-size Chunking）

**原理**：按固定字符数/Token数直接切。

**优点**：实现最简单，性能最好。

**缺点**：可能把一句话从中间切断，破坏语义完整性。

**适用场景**：对语义完整性要求不高的场景，或者文档本身结构就很规整。

---

### 2. 递归分块（Recursive Chunking）

**原理**：优先按大分隔符切，太大再按小分隔符切。

**分隔符优先级**：段落("\n\n") > 换行("\n") > 空格(" ") > 硬切

**优点**：尽量保持语义完整，实现不算复杂。

**缺点**：在没有明显分隔符的情况下（如长代码块），还是会硬切。

**适用场景**：通用场景，本项目用的就是这个。

---

### 3. 语义分块（Semantic Chunking）

**原理**：用 embedding 模型计算相邻句子/段落之间的语义相似度，在相似度突然下降的地方切分。

**流程**：
1. 把文本拆成小单元（句子或短句）
2. 每个单元生成 embedding
3. 计算相邻单元的余弦相似度
4. 相似度低于阈值 → 在这里切分

**示例**：
```
句子1: "LangGraph 是一个状态图框架。" → embedding1
句子2: "它用于构建有状态的工作流。" → embedding2
句子3: "Python 是一种编程语言。" → embedding3

sim(1,2) = 0.85（高，同主题）→ 不切
sim(2,3) = 0.42（低，换主题）→ 在这里切

结果：
chunk1: "LangGraph 是一个状态图框架。它用于构建有状态的工作流。"
chunk2: "Python 是一种编程语言。"
```

**优点**：语义完整性最好，不会把相关内容拆开。

**缺点**：需要调用 embedding 模型，比其他策略慢。

**适用场景**：高质量文档、语义连贯性要求高的场景。

**实现方式**：
- `langchain_experimental.text_splitter.SemanticChunker`
- 自己用 embedding 模型实现

---

### 4. 结构化分块（Structured Chunking）

**原理**：按文档的结构化元素切分。

**常见的结构化元素**：

| 格式 | 结构化元素 | 切分方式 |
|------|-----------|---------|
| **Markdown** | 标题（#、##、###） | 按标题级别切 |
| **HTML** | 标签（`<h1>`、`<p>`、`<div>`） | 按标签切 |
| **PDF** | 章节、页码 | 按章节切 |
| **代码** | 函数、类、模块 | 按代码结构切 |

**示例**（Markdown）：
```markdown
# LangGraph 简介

LangGraph 是一个状态图框架。

## 核心概念

### StateGraph

StateGraph 是 LangGraph 的核心。

### Node

节点是工作流中的基本单元。
```

按标题切分后：
```
chunk1: "# LangGraph 简介\n\nLangGraph 是一个状态图框架。"
chunk2: "## 核心概念\n\n### StateGraph\n\nStateGraph 是 LangGraph 的核心。"
chunk3: "### Node\n\n节点是工作流中的基本单元。"
```

**优点**：保持文档的结构层次，检索结果更容易理解。

**缺点**：需要解析文档结构，实现较复杂。

**适用场景**：结构化文档（Markdown、HTML、PDF）。

---

### 5. 段落分块（Paragraph Chunking）

**原理**：按段落切分（段落之间通常有两个换行符）。

**优点**：天然的语义边界，实现简单。

**缺点**：段落大小不一，有的很短，有的很长。

**适用场景**：新闻、论文、博客等段落清晰的文档。

---

### 6. 滑动窗口分块（Sliding Window Chunking）

**原理**：用固定大小的窗口，按固定步长滑动切分。

**示例**：
```
文本: "ABCDEFGHIJKLMNOPQRSTUVWXYZ"
window_size = 5, step = 2

chunk1: ABCDE
chunk2: CDEFG
chunk3: EFGHI
...
```

**优点**：覆盖所有内容，不会遗漏任何细节。

**缺点**：冗余度高，同一个内容会出现在多个 chunk 中。

**适用场景**：需要确保每个细节都被覆盖的场景，如法律文档、合同。

---

### 7. 基于实体的分块（Entity-based Chunking）

**原理**：按实体边界切分，确保同一实体的信息在同一个 chunk 中。

**流程**：
1. 用 NER（命名实体识别）提取实体
2. 确保同一实体的所有信息在同一个 chunk 中
3. 在实体边界处切分

**示例**：
```
文本: "LangGraph 是 LangChain 团队开发的。它支持 Python 和 JavaScript。"

实体: LangGraph, LangChain, Python, JavaScript

chunk1: "LangGraph 是 LangChain 团队开发的。"（包含 LangGraph 和 LangChain）
chunk2: "它支持 Python 和 JavaScript。"（包含 Python 和 JavaScript）
```

**优点**：适合知识密集型文档，确保相关实体的信息在一起。

**缺点**：需要实体识别模型，实现复杂。

**适用场景**：百科全书、技术文档、法律文档。

---

### 8. 分层分块（Hierarchical Chunking）

**原理**：同时做粗分和细分，建立分层索引。

**流程**：
1. **粗分**：把文档切成大 chunk（如 2000 字符）
2. **细分**：每个大 chunk 再切成小 chunk（如 512 字符）
3. 检索时：先检索大 chunk，再检索对应的小 chunk

**优点**：兼顾全局上下文和细节，提高检索精度。

**缺点**：存储开销大，需要维护两层索引。

**适用场景**：长文档、需要多粒度理解的场景。

---

## 三、如何选择分块策略？

| 场景 | 推荐策略 |
|------|---------|
| **通用场景** | 递归分块 |
| **高质量文档** | 语义分块 |
| **结构化文档（MD/HTML）** | 结构化分块 |
| **代码仓库** | 按函数/类分块 |
| **新闻/论文** | 段落分块 |
| **法律/合同** | 滑动窗口分块 |
| **百科/知识库** | 基于实体的分块 |
| **长文档** | 分层分块 |

---

## 四、面试怎么讲？（30秒版）

> 分块策略有很多种，每种适合不同场景。最基础的是固定大小分块，简单但可能切断语义；递归分块优先按段落、换行等分隔符切，尽量保持语义完整，是通用场景的首选；语义分块用 embedding 模型找语义边界，语义完整性最好，但需要调用模型比较慢；结构化分块按标题、章节等结构切，适合 Markdown、HTML 等结构化文档；段落分块按段落切，天然语义边界；滑动窗口分块用固定步长滑动，覆盖所有内容但冗余度高；基于实体的分块按实体边界切，适合知识密集型文档；分层分块同时做粗分和细分，兼顾全局和细节。我们项目用的是递归分块，因为它在语义完整性和实现复杂度之间取得了平衡。

## 五、面试高频追问

### Q1：语义分块和递归分块有什么区别？

递归分块是基于**字符层面的分隔符**（段落、换行、空格），是"规则驱动"的；语义分块是基于**语义层面的相似度**，是"数据驱动"的。递归分块可能把同一个主题的内容拆开（因为段落太长），而语义分块不会——它会在语义边界处切分。

### Q2：什么时候用分层分块？

当文档很长（如一本书、一篇长论文），需要同时理解全局上下文和细节时。分层分块先检索大 chunk 找到相关章节，再在该章节内检索小 chunk 找到具体细节，这样既不会因为 chunk 太小丢失上下文，也不会因为 chunk 太大包含太多噪声。

### Q3：分块大小怎么确定？

分块大小取决于多个因素：
- **模型上下文窗口**：不能超过 LLM 的上下文限制
- **语义完整性**：要保证一个 chunk 能包含完整的语义单元（一个段落、一个知识点）
- **检索精度**：chunk 越小，embedding 越精准，但可能丢失上下文
- **经验值**：512-1024 字符是常用范围，具体要看文档类型

### Q4：代码文件怎么分块？

代码文件应该按**语法结构**分块，而不是按字符数：
- Python：按函数、类、模块切
- JavaScript：按函数、类、模块切
- 通用：按代码块切（函数定义、类定义、注释块等）

这样可以保证代码的语义完整性，检索时能找到完整的函数或类。
