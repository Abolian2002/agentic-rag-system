# Chroma 稠密向量检索详解（面试学习笔记）

本文档详细讲解 retrieve 节点中的 Chroma 稠密向量检索技术，包括核心概念、完整流程、算法原理和面试要点。

---

## 一、什么是"稠密向量检索"？

BM25 是**词对词的精确匹配**——文档里必须有这个词才算数。但自然语言里，"苹果手机"和"iPhone"说的是一回事，BM25 就匹配不上。

**稠密向量检索的核心思想**：把每段文字转换成一个**高维向量（embedding）**，语义相近的文字，它们的向量在空间中距离也近。这样，"苹果手机"和"iPhone"的向量会很接近，就能匹配上。

### 术语解释

- **"稠密"**：向量的每个维度都有实数值（比如 768 维、1536 维），不像 BM25 的稀疏向量大部分维度是 0
- **"向量检索"**：拿 query 的向量去文档向量库里找"最近的邻居"
- **"语义匹配"**：不依赖精确词汇，而是理解文字表达的意思

---

## 二、完整流程（两大阶段）

### 阶段 A：索引阶段（文档加载时做一次）

就像 BM25 要建倒排索引，向量检索也要建"向量索引"。

#### 步骤 1：文档切分

把 PDF/DOCX 切成 chunk（本项目默认每段 512 个 token，有重叠）。

#### 步骤 2：每个 chunk 生成 Embedding

调用 Embedding 模型（本项目用的是 `nomic-embed-text`，通过 Ollama 运行），把每个 chunk 的文字变成一个高维向量。

**示例**（简化为 3 维，实际是 768 维）：

| chunk | 向量 |
|-------|------|
| chunk_1: "LangGraph 是一个状态图框架..." | [0.12, -0.34, 0.56] |
| chunk_2: "BM25 是稀疏检索算法..." | [-0.78, 0.21, -0.45] |
| chunk_3: "向量检索基于语义相似度..." | [0.65, -0.12, 0.33] |

#### 步骤 3：存入 Chroma 向量数据库

Chroma 把这些向量持久化存储在磁盘上，同时构建一个**近似最近邻（ANN）索引**，方便查询时快速找到最近的向量。

Chroma 默认用的是 **HNSW（Hierarchical Navigable Small World，分层可导航小世界）** 索引结构——这是目前最流行的 ANN 算法之一。

---

### 阶段 B：查询阶段（用户每次提问时）

#### 步骤 1：用户问题生成 Embedding

把用户的问题也用**同一个 embedding 模型**转成向量。

> ⚠️ **关键**：query 和 document 必须用**同一个模型**生成 embedding，不然不在同一个向量空间里，相似度没有意义。

#### 步骤 2：在 Chroma 中做相似度搜索

拿 query 向量去向量库里找"最相似的 Top-K 个文档向量"。

**相似度怎么算？** 用**余弦相似度（Cosine Similarity）**：

```
cos_sim(A, B) = (A · B) / (||A|| × ||B||)
```

就是两个向量的夹角的余弦值：
- 夹角越小 → 越相似 → 余弦值越接近 1
- 夹角越大 → 越不相似 → 余弦值越接近 0 或 -1

取值范围：**[-1, 1]**，但文本 embedding 通常都是正数，所以实际范围是 [0, 1]。

#### 步骤 3：返回 Top-K 结果

按相似度从高到低排序，返回最相关的 K 个 chunk（本项目默认 K=4，重试时扩到 6）。

## 三、为什么叫"近似"最近邻（ANN）？

### 暴力搜索的问题

理论上最准确的是**暴力搜索（Brute Force）**：拿 query 向量和库中每一个文档向量都算一遍余弦相似度，取前 K 个。但文档多了根本算不完——

| 文档数量 | 暴力搜索耗时 | 现实可行性 |
|---------|------------|-----------|
| 10,000 | 可接受 | ✅ |
| 100,000 | 较慢 | ⚠️ |
| 1,000,000 | 太慢 | ❌ |

### ANN 的解决方案

**ANN（Approximate Nearest Neighbor）** 就是用一些数据结构和算法，**牺牲一点点精度，换取几十上百倍的速度提升**。

### 常见的 ANN 算法

| 算法 | 思路 | 代表系统 |
|------|------|---------|
| **HNSW** | 构建多层小世界图，从顶层粗跳到细层精搜 | Chroma 默认、Faiss IVF-HNSW |
| **IVF** | 聚类分桶，先找最近的几个桶再在桶内暴力搜 | Faiss、Milvus |
| **PQ** | 向量量化压缩，用近似表示减少内存 | Faiss |

### HNSW 的工作原理（简化版）

1. **构建阶段**：
   - 在向量空间中随机选一些向量作为"导航点"
   - 每个向量连接到它附近的几个向量（形成小世界图）
   - 构建多层结构：顶层稀疏、底层稠密
   
2. **查询阶段**：
   - 从顶层开始，沿着连接快速跳到最近的区域
   - 逐层往下精搜，最终找到真正近的邻居

**特点**：查询快、构建也快、召回率高，是目前的主流选择。

## 四、Chroma 具体做了什么？

Chroma 是一个**轻量级向量数据库**，它帮你封装了以下事情：

| Chroma 的能力 | 说明 |
|--------------|------|
| **持久化存储** | 向量存在磁盘上，重启不丢（FAISS 是纯内存的，要自己存） |
| **Embedding 集成** | 内置多种 embedding 函数（OpenAI、HuggingFace、Ollama 等） |
| **ANN 索引** | 默认 HNSW，自动构建和维护 |
| **元数据过滤** | 可以按来源文件、chunk 编号等元数据过滤检索范围 |
| **增删改查** | 支持文档级别的 CRUD，方便做增量索引 |
| **集合（Collection）** | 不同知识库可以放不同 collection，互不干扰 |

### 为什么选 Chroma 而不是 FAISS？

| 维度 | Chroma | FAISS |
|------|--------|-------|
| 持久化 | ✅ 自带 | ❌ 需要自己实现 |
| 增量索引 | ✅ 支持 | ❌ 不支持，需全量重建 |
| 元数据 | ✅ 原生支持 | ❌ 需要额外存储 |
| CRUD | ✅ 完整 | ❌ 只支持添加，不支持删除更新 |
| 易用性 | ✅ API 友好 | ❌ C++ 接口，Python 绑定有限 |

## 五、和 BM25 的核心区别

| 维度 | BM25（稀疏） | Chroma 向量检索（稠密） |
|------|-------------|----------------------|
| **匹配依据** | 精确词汇匹配 | 语义相似度 |
| **索引结构** | 倒排索引 | ANN 向量索引（HNSW） |
| **能处理同义词** | ❌ 不能 | ✅ 能 |
| **能处理专有名词/代码** | ✅ 精确 | ⚠️ 较差（容易混淆相似名） |
| **查询速度** | 极快（O(1) 查词） | 快（ANN 近似搜索） |
| **可解释性** | 高（知道哪个词命中的） | 低（黑盒向量） |
| **依赖模型** | 不依赖（纯统计） | 依赖 embedding 模型质量 |

## 六、混合检索的权重配置

在本项目中，BM25 和向量检索通过 `EnsembleRetriever` 加权融合：

| 检索器 | 权重 | 原因 |
|--------|------|------|
| BM25 | 0.35 | 语义匹配更重要，但精确词匹配不可替代 |
| Chroma 向量 | 0.65 | RAG 场景下，语义匹配覆盖更多改写问法 |

**融合公式**（简化版）：

```
最终分数(D) = 0.35 × BM25分数(D) + 0.65 × 向量相似度(D)
```

## 七、面试怎么讲？（30秒版）

> 稠密向量检索的核心是把文本转成高维向量用语义相似度匹配，而不是精确词匹配。流程分两步：索引阶段把每个 chunk 用 embedding 模型转成向量，存到 Chroma 里构建 HNSW 近似最近邻索引；查询阶段把用户问题也转成向量，在向量库里做余弦相似度搜索，返回 Top-K 最相似的文档。
>
> 之所以用近似最近邻（ANN）而不是暴力搜索，是因为文档量大了之后全库比对太慢，ANN 牺牲一点点精度换取几十上百倍的速度提升。Chroma 默认用的是 HNSW 算法，通过分层小世界图实现高效的近似搜索。

## 八、面试高频追问

### Q1：余弦相似度和欧氏距离有什么区别？

- **余弦相似度**：看方向，不管长度（向量长短不影响，只看夹角）——文本 embedding 通常已经归一化，所以用余弦
- **欧氏距离**：看空间中的绝对距离——如果向量长度有意义（比如像素值），用欧氏
- 文本检索用余弦相似度，因为我们关心的是"语义方向"是否一致，不是向量模长

### Q2：Embedding 模型的维度是啥意思？越高越好吗？

维度是向量的长度，比如 768 维、1536 维。
- 维度越高 → 能表达的语义信息越丰富 → 精度可能越高
- 但维度越高 → 存储越大、计算越慢、ANN 精度下降（维度灾难）
- 所以是个 trade-off，个人知识库 768 维（nomic-embed-text）就足够了

### Q3：为什么 BM25 和向量检索要混合用？

因为两者互补——BM25 抓精确词汇（专有名词、代码、数字），向量检索抓语义相似（同义词、改写）。混合后召回率比单独用任何一种都高，这就是 Hybrid Search 的价值。

### Q4：Chroma 和 Milvus/Pinecone 有什么区别？

| 维度 | Chroma | Milvus / Pinecone |
|------|--------|-------------------|
| 定位 | 轻量、嵌入式、本地 | 企业级、云服务 |
| 部署方式 | 单进程、无依赖 | 分布式集群 |
| 适用场景 | 个人项目、小团队 | 大规模生产环境 |
| 成本 | 零成本 | 付费 |

本项目选 Chroma 是因为它足够轻量，单文件就能运行，适合个人知识库场景。

### Q5：Embedding 模型是怎么训练出来的？

（简要回答）Embedding 模型通常用对比学习（Contrastive Learning）训练——让语义相近的文本对在向量空间中靠近，语义不同的文本对远离。常见训练任务：
- **Next Sentence Prediction（NSP）**：判断两段话是否连续
- **对比学习**：正样本对拉近，负样本对推开

## 九、核心概念速记

| 术语 | 一句话解释 |
|------|-----------|
| Embedding | 文本 → 高维实数向量的转换 |
| 稠密向量 | 每个维度都有值的向量 |
| 余弦相似度 | 衡量两个向量方向是否一致（[-1, 1]） |
| ANN | 近似最近邻搜索，牺牲精度换速度 |
| HNSW | 分层小世界图，Chroma 默认的 ANN 算法 |
| Hybrid Search | BM25（精确）+ 向量检索（语义）混合 |
